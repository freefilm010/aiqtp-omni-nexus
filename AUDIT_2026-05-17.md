## 🔥 Critical findings (act now)

### C1 — `.github/workflows/get-secrets.yml` leaks Supabase API keys to public Actions logs
**File:** `.github/workflows/get-secrets.yml`
**Risk:** The workflow calls `https://api.supabase.com/v1/projects/<ref>/api-keys` and pipes the response to stdout. On a **public** repository, GitHub Actions logs are world-readable. The Supabase Management API `/api-keys` endpoint returns the project's **service_role** key in cleartext — which bypasses RLS and grants full DB access.
**Evidence of execution:** Commits `4c21592` ("Add temporary workflow to retrieve project config") and `85102bd` ("Update workflow trigger to run on push") suggest this has already run.
**Action:**
1. Delete `.github/workflows/get-secrets.yml`
2. Rotate `SUPABASE_ACCESS_TOKEN`
3. Rotate the Supabase **service_role** JWT (Supabase dashboard → Settings → API → "Generate new JWT secret" — invalidates anon + service_role; you'll need to update all consumers)
4. Manually inspect Actions logs for the `get-secrets` workflow runs and consider deleting those runs

### C2 — Live trading is hot and depends on safeguards in `trading_worker.py`
**File:** `render.yaml` line 33-35 (`ALPACA_PAPER_MODE: "false"`), `RUNBOOK.md` line 110-114
**Risk:** Real money is on the line. RUNBOOK claims guardrails: max 2% NAV/order, 5 orders/hr per user, symbol whitelist. None of these have been verified to actually exist in the code or to be exercised by tests.
**Action:** Audit `core-brain/trading_worker.py` + `core-brain/real_strategy.py` for the claimed guardrails. Confirm the kill switch UI (`MasterKillSwitch.tsx`) is wired to an actually-respected backend flag.

### C3 — Observer-mode override in `CLAUDE.md` blocks AI agents
**File:** `CLAUDE.md` lines 1-3, introduced by commit `638ef39` ("Freeze Claude Code (observer mode)")
**Risk:** First three lines tell any Claude Code instance to refuse to write/edit/run anything. If you want AI assistance on this repo (which you do — you're asking for it now), this header neutralizes it.
**Action:** Remove the observer-mode header (keep the architecture content below).

## ⚠️ High-priority findings

### H1 — `.env.development` and `.env.production` are committed
**Files:** `.env.development`, `.env.production`
**What leaked:**
- `VITE_PAYMENTS_CLIENT_TOKEN=pk_live_51SmyGB…` (Stripe **publishable** — safe but should be in Vercel)
- `VITE_PAYMENTS_CLIENT_TOKEN=pk_test_51TQmjm…` (Stripe test publishable — same)
- `VITE_SUPABASE_PUBLISHABLE_KEY=eyJ…` (Supabase **anon** JWT — safe but should be in Vercel)
- Supabase project URL + project ID (not secret)

**Risk:** None of these are independently exploitable (they're "publishable" / "anon" by design, RLS protects you). But:
1. Violates CLAUDE.md rule #3 ("never hardcode secrets — use env vars")
2. `.gitignore` patterns (`.env`, `.env.local`, `.env.*.local`) **do not match** `.env.development` / `.env.production` — gap in defense
3. If anyone later edits these files and adds a real secret, gitleaks won't catch it because they're already tracked

**Action:**
1. Update `.gitignore` to `.env*` (then re-allow `.env.example` with `!.env.example`)
2. `git rm --cached .env.development .env.production`
3. Move values to Vercel env vars (already partially done per CLAUDE.md)

### H2 — Anon JWT hardcoded in **6 separate workflow files**
**Files:** `.github/workflows/omni-bot.yml`, `supabase-backup.yml`, `supabase-keepalive.yml`, `sync-monitor.yml`, plus `.env.*`, `.gitleaksignore`
**Risk:** Same anon JWT pasted in 6+ places. If you ever need to rotate it (e.g., due to C1), you'll forget one and break monitoring.
**Action:** Move to a single GitHub Actions secret (`SUPABASE_ANON_KEY`) and reference it as `${{ secrets.SUPABASE_ANON_KEY }}`.

### H3 — CLAUDE.md TODO list contains unresolved deployment blockers

- [ ] Broker API keys (TRADIER/BINANCE/KRAKEN/IBKR) not in Render → those endpoints will 401 in production
- [ ] `STRIPE_SECRET_KEY` not in Render → checkout/webhook flow is broken
- [ ] Pending SQL migrations not applied → `scripts/apply-new-migrations.sql` is unapplied

Will create duplicate dependency-bump PRs. Pick one.
